# AI 自动化测试需求说明（需求确认版）

**版本**: v1.0  
**更新时间**: 2026-04-30  
**状态**: 需求确认 / 方案约束版

---

## 1. 背景

当前系统已经具备“代码分析、测试用例生成、Agent 工具调用、执行记录保存”的基础能力，但现阶段的脚本执行路径仍然是“让模型生成完整 Playwright 脚本，再落盘执行”。这会带来两个问题：

1. 模型输出如果包含 Markdown 代码块，会直接导致脚本执行失败。
2. 纯脚本执行不适合你想要的“像人在普通浏览器里点按钮、输入、查看页面变化”的效果。

因此，本需求的核心方向是：**Claude CLI 负责初版用例生成，Agent 负责基于 function call 的浏览器交互执行**。

---

## 2. 目标

本系统要支持如下完整流程：

1. 用户先提供项目仓库信息，包括 GitLab / GitHub 地址、账号、密码或 Token。
2. 系统将仓库 clone 到本地指定目录。
3. 用户再提供测试功能和测试目标页面或接口。
4. Claude CLI 读取本地代码，分析实现逻辑，生成初版测试用例。
5. Agent 与用户进行多轮对话，继续补充、修改、确认用例。
6. 用户点击执行后，Agent 通过 function call 真正操作浏览器完成测试。
7. 执行结果、步骤日志、截图和耗时保存到数据库，并在前端展示。

### 目标边界

- 只需要普通桌面浏览器界面。
- 不需要真手机。
- 不需要移动端真实设备。
- 需要的是“浏览器可见、步骤可追踪、点击可解释”的执行体验。

---

## 3. 非目标

以下内容本期不作为必须项：

1. 真机自动化。
2. 云手机或设备农场接入。
3. 复杂移动端适配测试。
4. 完全替代现有的脚本执行模式。

现有脚本模式可以保留为兜底方案，但不应作为主执行路径。

---

## 4. 用户流程

### 4.1 项目初始化

用户创建项目时，需要提供：

- 项目名称
- 被测项目 Git 地址
- Git 账号 / 凭证
- 本地 clone 路径
- 被测系统访问地址
- 被测系统账号 / 密码（如需要）

系统完成：

1. clone 仓库。
2. 记录本地仓库路径。
3. 为后续代码分析准备上下文。

### 4.2 初版用例生成

用户输入：

- 测试功能
- 测试页面 / 接口

Claude CLI 根据本地仓库代码进行分析，生成初版用例，重点包括：

- 页面元素或接口路径定位
- 正常流步骤
- 校验点
- 异常场景补充建议

### 4.3 对话式调整

Agent 接收初版用例后与用户对话：

- 补充边界条件
- 补充异常场景
- 调整步骤顺序
- 调整预期结果

用户确认后，用例状态进入“可执行”。

### 4.4 自动执行

用户点击执行后，Agent 使用浏览器工具完成：

- 打开页面
- 输入内容
- 点击按钮
- 等待元素出现
- 读取页面文本
- 截图记录
- 上报结果

执行结束后，系统返回：

- 执行状态
- 执行耗时
- 步骤日志
- 截图列表
- 错误信息或成功摘要

1. Agent 方向是对的，因为它已经支持工具调用。
2. 脚本模式不适合作为主路径，因为它会受模型输出格式干扰。
3. 前端展示还不够直观，目前只能看日志，不能很好地看步骤和截图。

## 6. 建议的实现方式

### 6.1 Claude CLI 负责初版生成

Claude CLI 的角色只做一件事：

- 读取 clone 后的仓库代码
- 找到相关页面、接口、组件、路由
- 生成初版测试用例

不建议让 Claude CLI 直接负责最终点击执行。

### 6.2 Agent 负责 function call 执行

Agent 的职责是：

- 根据已确认的用例一步一步操作浏览器
- 通过 function call 执行点击、输入、等待、读取文本、截图
- 每一步都记录到执行日志

### 6.3 浏览器使用普通桌面模式

本需求明确采用普通桌面浏览器，不使用真手机。

推荐实现方式有两种：

1. 默认无头执行：适合后台批量执行。
2. 可见窗口执行：适合调试和观察点击过程。

系统中已经有 `agent_headless` 这样的配置项，可以把它作为是否显示浏览器窗口的开关。

### 6.4 日志展示升级

前端建议从“纯文本日志”升级为：

- 步骤时间线
- 当前执行状态
- 每一步的 tool call 记录
- 截图缩略图
- 失败步骤高亮

如果想更接近“看到它在点”，可以继续做：

- 执行过程中定时刷新截图
- 或者直接打开可见浏览器窗口进行观察

## 7. 需求拆分

### 7.1 必须实现

- 项目仓库 clone 和管理
- Claude CLI 初版用例生成
- Agent 多轮对话调整
- Agent function call 执行
- 普通桌面浏览器交互
- 执行结果落库
- 前端执行详情查看

- 更丰富的浏览器定位方式，例如支持文本、角色、标签定位
- 步骤级截图
- 执行过程回放
- 失败时自动保存最后一屏

### 7.3 后续增强

- WebSocket 实时推送执行状态
- 更完整的报告导出
- 用例版本管理
- 权限和项目隔离

## 8. 当前问题与处理建议

### 问题 1：脚本中带了 Markdown 代码块会报错

现象是模型返回的内容被原样写成脚本执行，导致 `SyntaxError`。

建议处理方式：

1. 如果继续保留脚本模式，必须先清洗输出内容。
2. 更推荐把主执行路径切到 Agent function call。

### 问题 2：前端只看得到文本日志

当前执行页只能看到日志字符串，不利于理解浏览器实际操作过程。

建议补充：

1. 步骤列表。
2. 截图展示。
3. 每一步工具调用详情。

### 问题 3：浏览器默认无头

如果你想观察执行过程，需要把浏览器从无头模式切成可见模式。

建议：

- 默认保留无头模式用于正式执行。
- 调试时允许开启可见窗口。

---

## 9. 验收标准

满足以下条件时，可认为这一版需求达成：

1. 用户能通过仓库地址和账号完成项目初始化。
2. Claude CLI 能基于 clone 后代码生成初版测试用例。
3. Agent 能对初版用例进行多轮修改确认。
4. Agent 能在普通桌面浏览器里完成点击、输入、等待和断言。
5. 执行记录中可以看到步骤日志和截图。
6. 前端能够查看执行过程和结果。
7. 不依赖真手机也能完成完整自动化测试流程。

---

## 10. 推荐下一步

如果后续要继续实现，建议优先顺序如下：

1. 先把执行主路径切到 Agent function call。
2. 再补前端执行详情页。
3. 最后再考虑实时可视化和更丰富的步骤回放。

---

## 11. 结论

这套需求的核心不是“继续生成可执行脚本”，而是“用 Claude CLI 做代码理解和用例生成，用 Agent 做真正的浏览器操作执行”。

当前版本明确采用：**普通桌面浏览器 + function call + 执行日志 + 截图记录**，不使用真手机。
