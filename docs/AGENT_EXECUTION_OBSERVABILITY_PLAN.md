# Agent 执行过程可视化界面规划

## 1. 文档目的

这份文档用于规划 My_Test 项目后续的 Agent 执行过程可视化界面。

目标不是马上改代码，而是先基于当前项目已经具备的能力，明确后续应该把“Agent 执行测试用例时到底做了什么”展示成什么样，方便后续按阶段实现。

你提出的核心诉求是：

- 点击 Agent 执行测试用例后，页面上可以看到 Agent 打开了哪个页面
- 可以看到 Agent 点击了什么元素
- 可以看到每一步执行后有什么反应
- 可以看到 Agent 的具体操作过程，而不是只看最终成功或失败

这本质上是在做“执行结果页”升级为“执行过程观测页”。

## 2. 当前项目内容梳理

根据目前项目结构和代码实现，My_Test 是一个 Django + Vue 3 的 AI 自动化测试平台，主要由下面几部分组成：

### 2.1 后端能力

- Django 提供项目、测试用例、执行记录、系统设置等接口
- 支持两种执行模式：script 和 agent
- Agent 执行结果会写入 ExecutionRecord
- 执行过程中已经会保存结构化数据，而不只是纯文本日志

当前和本需求直接相关的后端对象：

- Project：测试项目
- TestCase：测试用例
- ExecutionRecord：执行记录
- Screenshot：执行截图持久化记录

ExecutionRecord 当前已经具备这些关键字段：

- status：执行状态
- execution_mode：执行模式
- tool_calls_count：工具调用次数
- ai_model：使用的模型
- log：执行日志
- error_message：错误信息
- screenshots：截图列表
- step_logs：结构化步骤日志
- agent_response：Agent 原始响应

这说明后端不是“没有数据”，而是“数据已经有了，但前端还没有把它组织成清晰的过程界面”。

### 2.2 前端能力

当前前端里，和执行记录最相关的页面有两个：

- frontend/src/views/Executions.vue
- frontend/src/views/ProjectDetail.vue

当前已经具备的界面能力：

- 有全局“执行记录”列表页
- 有项目详情页中的“执行记录”入口
- 有执行详情弹窗
- 有原始日志弹窗
- 有截图展示组件 ScreenshotGallery
- 有 step_logs 的时间线基础展示

也就是说，现在系统不是完全没有展示，而是展示层级还不够合理，信息虽然在，但用户很难一眼看懂 Agent 做了什么。

## 3. 当前已具备的数据基础

从现有代码看，后续做“Agent 过程可视化界面”已经有比较好的基础。

### 3.1 已有的步骤数据

当前 step_logs 已经包含这些信息：

- step_num：步骤序号
- action：动作类型
- target：目标对象
- result：执行结果摘要
- timestamp：时间戳
- screenshot_path：该步骤关联截图

这些字段已经足够支撑第一版“过程时间线”。

### 3.2 已有的截图数据

当前系统已经支持：

- 在执行过程中生成截图
- 把截图路径写入 execution 记录
- 通过后端接口安全读取截图
- 在前端用 ScreenshotGallery 展示截图

这意味着后续不需要从零设计截图体系，只需要重新组织截图和步骤的对应关系。

### 3.3 已有的操作来源数据

Agent 模式执行时，底层会记录 tool_calls_log，并最终整理成 step_logs。

从现有逻辑看，已经能够区分一些关键动作，例如：

- 打开页面
- 点击元素
- 输入内容
- 截图

这正是你想看到的“Agent 做了什么”的来源。

## 4. 当前界面存在的问题

虽然数据已经有了，但当前页面存在几个明显问题。

### 4.1 问题一：执行记录偏结果导向，不偏过程导向

当前主列表重点展示的是：

- 状态
- 耗时
- 工具调用次数
- 执行时间

这更像“报表页”，而不是“回放页”。

用户最关心的问题“它刚才点了什么、打开了哪里、为什么失败”没有放在最前面。

### 4.2 问题二：信息被拆在多个弹窗里

当前至少拆成了：

- 详情弹窗
- 日志弹窗
- 图片预览弹窗

用户要来回点多个入口，信息流是断的，很难连续理解一次执行过程。

### 4.3 问题三：步骤时间线信息密度不够

当前时间线虽然有 step_logs，但展示层级还不够：

- 哪一步打开了新页面，不够突出
- 哪一步点击了哪个元素，不够结构化
- 点击后页面发生了什么，没有单独强调
- 失败点没有形成明显的视觉聚焦

### 4.4 问题四：截图和步骤的关系不够强

现在截图能看，但更像“附属资源”，不是“步骤证据”。

理想状态应该是：

- 看步骤时就能看到对应截图
- 看截图时能知道它属于第几步
- 出错步骤的截图优先展示

### 4.5 问题五：全局页和项目页有重复展示逻辑

Executions.vue 和 ProjectDetail.vue 都有执行详情相关展示。

这会带来两个问题：

- 后续改一个地方容易漏另一个地方
- 用户在不同入口看到的体验可能不一致

## 5. 后续界面设计目标

后续界面需要从“执行结果列表”升级成“执行过程观察台”。

建议把目标拆成四层：

### 5.1 第一层：快速摘要

用户打开某次执行后，第一眼就能看到：

- 这次执行的是哪个项目、哪个测试用例
- 当前状态是什么
- Agent 还是 script 模式
- 总耗时多少
- 总共执行了多少步
- 是否有异常或失败

### 5.2 第二层：过程时间线

核心区域应该是时间线或步骤流，逐步展示：

- 第 1 步打开了哪个页面
- 第 2 步点击了哪个按钮或元素
- 第 3 步输入了什么内容
- 第 4 步页面返回了什么结果
- 第 5 步为什么失败或通过

### 5.3 第三层：视觉证据

每个关键步骤最好能关联：

- 截图
- 页面标题或 URL
- 元素定位信息
- 执行结果摘要

这样页面就不只是“文字日志”，而是“有证据的过程追踪”。

### 5.4 第四层：调试信息

对普通用户隐藏细节，对调试用户保留入口：

- 原始日志
- 原始 agent_response
- 工具调用总数
- 错误堆栈

这部分不应该抢主界面，但必须可展开查看。

## 6. 推荐的界面方案

建议后续采用“列表页 + 统一详情面板”的结构，而不是继续堆更多弹窗。

### 6.1 方案总览

建议保留执行记录列表页作为入口，但执行详情改成更强的统一详情视图。

推荐结构：

- 左侧或上方：执行记录列表
- 右侧或详情区域：当前选中执行的完整过程视图

如果后续空间允许，也可以升级为独立执行详情页。

### 6.2 推荐页面分区

#### A. 执行摘要区

放在顶部，展示：

- 用例名称
- 项目名称
- 执行状态
- 执行模式
- 总耗时
- 工具调用次数
- 开始时间
- 失败原因摘要

#### B. 过程时间线区

放在页面主体，按步骤卡片展示：

- 步骤编号
- 动作类型
- 目标对象
- 时间点
- 执行结果
- 当前步骤截图

动作类型建议在视觉上区分：

- 打开页面
- 点击元素
- 输入内容
- 等待结果
- 截图
- 断言
- 报告结果

#### C. 页面状态区

如果当前数据足够，建议补充展示：

- 当前页面 URL
- 页面标题
- 页面切换提示

这一块是后续增强项。第一版如果暂时没有完整数据，可以先留占位规划。

#### D. 截图证据区

建议两种展示方式并存：

- 每一步卡片内的小图预览
- 页面底部或右侧的完整截图画廊

这样既能看单步证据，也能快速扫全程。

#### E. 调试折叠区

默认折叠，需要时展开：

- 原始日志
- 原始 agent 响应
- 结构化 JSON
- 错误详情

## 7. 重点展示什么信息

围绕你的诉求，界面必须优先突出下面这些内容。

### 7.1 Agent 打开了哪个界面

需要突出显示：

- 页面 URL
- 页面标题
- 页面切换行为

如果当前 step_logs 里只有 target，没有单独的页面标题字段，第一版可以先展示：

- 导航目标地址
- 该步截图

后续再增强记录页面标题。

### 7.2 Agent 点击了什么界面元素

需要展示：

- 动作类型为“点击”
- 点击目标 selector 或元素描述
- 点击前后截图对比入口
- 点击结果摘要

如果 target 现在还是 selector 字符串，第一版可以直接展示 selector；
后续可以考虑补充更友好的元素名称。

### 7.3 页面有什么反应

每一步需要有“结果反馈”区域，例如：

- 成功打开页面
- 找到按钮并点击
- 输入成功
- 页面弹出错误提示
- 页面无响应
- 断言失败

也就是把 step_logs.result 的可读性做强，而不是只展示一段截断文本。

### 7.4 可以看到 Agent 的具体操作

建议把一次执行过程理解为“可回放操作流”：

- 进入页面
- 定位元素
- 点击
- 等待
- 输入
- 提交
- 观察结果
- 截图
- 输出结论

界面上要让用户不看原始日志，也能理解这条操作流。

## 8. 推荐的信息架构

### 8.1 全局执行页职责

全局执行页适合负责：

- 查看所有项目的执行历史
- 按状态、项目、模式筛选
- 快速定位失败执行
- 进入某次执行的详细过程

### 8.2 项目详情页职责

项目详情页适合负责：

- 查看当前项目下的执行记录
- 从测试用例列表直接跳到执行过程
- 强调和项目上下文的关系

### 8.3 统一执行详情组件职责

后续最好抽成统一的“执行详情视图”，供两个页面复用。

这个统一视图负责：

- 摘要头部
- 步骤时间线
- 截图联动
- 调试信息折叠区

这样可以避免两处重复维护。

## 9. 分阶段改进建议

建议不要一次做太重，分三期推进。

### 9.1 第一期：界面重组

目标：不改后端数据结构，先把现有信息展示清楚。

建议做的事：

- 统一执行详情展示结构
- 把详情弹窗升级成“摘要 + 时间线 + 截图 + 调试区”
- 强化 step_logs 的视觉层级
- 让截图和步骤一一对应
- 失败步骤高亮

这期就已经能明显提升体验。

### 9.2 第二期：补强过程信息

目标：让“页面发生了什么”更清晰。

建议补充的数据或展示能力：

- 页面标题
- 当前 URL 变化
- 点击前后状态说明
- 更清晰的动作分类
- 更友好的 target 展示文案

### 9.3 第三期：增强回放感

目标：让执行过程更接近可视化回放。

建议方向：

- 步骤自动播放
- 连续截图切换
- 失败步骤自动定位
- 过滤只看关键步骤
- 按动作类型筛选

## 10. MVP 建议

如果只做一个最小但有明显价值的版本，建议 MVP 包含：

- 执行摘要卡片
- 结构化步骤时间线
- 每步关联截图
- 失败步骤高亮
- 原始日志折叠查看

MVP 暂时可以不做：

- 真正的视频回放
- 点击前后差异分析
- 页面 DOM 快照
- 多屏并排对比

## 11. 对后端数据的判断

从当前项目实现来看，第一版规划不需要先改数据库结构。

原因是：

- step_logs 已经有步骤序号、动作、目标、结果、时间、截图路径
- screenshots 已经能直接展示
- log 和 error_message 也都在
- agent_response 可以作为深度调试数据备用

因此第一阶段更适合先做前端信息重组。

只有当你后续要求更强的“页面变化感知”时，才需要考虑后端额外记录：

- 页面标题
- 跳转前后 URL
- 元素可读名称
- 更明确的成功失败分类

## 12. 最终建议

建议把后续要做的重点明确为一句话：

把“执行记录页”升级成“Agent 执行过程观察页”，让用户无需阅读原始日志，也能看懂 Agent 打开了哪个页面、点击了什么元素、页面出现了什么反应，以及最终为什么成功或失败。

优先级建议如下：

1. 先统一执行详情界面的信息结构
2. 再强化步骤时间线和截图联动
3. 最后补足更细的页面上下文数据

## 13. 本次规划对应的项目现状结论

本项目并不是缺“执行过程数据”，而是缺“面向用户理解的执行过程界面”。

现有后端数据已经可以支撑第一版较完整的过程展示，所以后续改进重点应该放在：

- 页面结构重组
- 信息优先级调整
- 执行步骤可读性提升
- 截图与步骤联动

而不是先去重做执行引擎。
