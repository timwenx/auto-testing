## 一、AI 功能优化（6 个功能点）

### 功能 1.1：AI 测试用例生成

> 源文件：`core/ai_engine.py` → `generate_testcases()`
> 前端：`ProjectDetail.vue` → AI 生成对话框

#### 现状

- 用户在项目详情页点击"AI 生成"，输入需求描述后调用 `POST /api/ai/generate-testcase/`
- 后端调用 `generate_testcases()` 发送单次 prompt 给 Claude，要求返回 JSON 数组
- `_gather_code_context()` 收集项目代码上下文（目录树 + 关键词搜索代码片段）注入 prompt
- 前端收到 JSON 后弹出 `BatchTestCaseEditor` 让用户确认并保存
- **无重试机制**：API 调用失败直接抛异常，前端显示"AI 调用失败"
- **无进度反馈**：生成过程中前端只显示 loading，用户不知道 AI 在做什么
- **单次生成无交互**：用户无法在生成过程中与 AI 对话调整方向

#### 升级方案

1. **流式响应 (Streaming)**：将 `client.messages.create()` 改为 `client.messages.stream()`，通过 SSE/WebSocket 逐步推送生成进度给前端（如"正在分析需求..."、"正在生成第 3 条用例..."）
2. **智能重试**：对 429 限速和 500 服务端错误增加指数退避重试（最多 3 次），网络超时单独处理
3. **生成质量反馈**：生成完成后自动评估用例质量（完整性、覆盖度评分），在确认界面展示

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 生成体验 | 黑盒等待 30-60s，无反馈 | 实时看到 AI 思考过程，体感等待时间降低 |
| 失败处理 | 直接报错，需重试 | 自动重试 3 次，仅真正失败才报错 |
| 用例质量 | 无法评估 | 自动评分提示，用户可针对性补充 |

### 功能 1.2：AI 对话式用例调整

> 源文件：`core/ai_engine.py` → `adjust_testcases()`
> 前端：`ProjectDetail.vue` → AI 生成对话框内的多轮对话

#### 现状

- `adjust_testcases()` 接收对话历史 + 用户反馈 + 当前用例，返回修改后的完整 JSON 数组
- 前端通过 `conversation_history` 维护多轮对话上下文
- 每次调整都是**全量替换**：返回全部用例的完整 JSON，即使只改了一条
- 对话历史无上限，多轮后 token 消耗线性增长
- **无 diff 预览**：用户看不到改了什么，只能对比整体 JSON

#### 升级方案

1. **增量修改**：调整 prompt 让 AI 返回 `{action: "add/modify/delete", target: "用例名", changes: {...}}` 的增量格式，前端做合并，减少 token 浪费和全量替换风险
2. **Diff 预览**：前端对比调整前后的用例列表，高亮显示新增/修改/删除的用例，用户确认后才生效
3. **对话历史滑动窗口**：保留最近 10 轮对话，超出部分摘要压缩，控制 token 消耗

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 修改可见性 | 不知道改了什么 | 清晰的 diff 视图，修改处高亮 |
| Token 消耗 | 每轮完整用例 JSON + 全部历史 | 增量修改 + 历史窗口，降低 50%+ |
| 操作信心 | 盲目确认 | 确认前可审查每处改动 |

### 功能 1.3：AI 执行结果分析

> 源文件：`core/ai_engine.py` → `analyze_execution()`

#### 现状

- 执行完成后可调用 `POST /api/ai/analyze-result/` 分析失败原因
- 返回结构化 JSON：`summary, root_cause, suggestion, severity, is_flaky`
- **需手动触发**：用户需在执行详情中主动点击"分析"按钮
- **分析结果无持久化**：返回后仅在弹窗中展示，关闭即丢失
- **无批量分析**：不能对一批执行结果做整体分析

#### 升级方案

1. **自动分析**：Agent 执行失败（failed/error）时自动调用分析，结果存入 `ExecutionRecord` 新增的 `analysis_result` JSONField
2. **分析历史**：前端展示分析结果卡片（严重程度标签 + 根因摘要 + 建议），支持查看历史分析
3. **批量分析摘要**：对项目维度的所有失败执行做聚合分析，生成"失败 Top 原因"、"高频 Flaky 用例"等统计

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 分析时机 | 手动触发 | 失败自动分析 + 手动可触发 |
| 结果留存 | 关闭即丢 | 持久化存储，随时查看 |
| 全局洞察 | 无 | 项目级失败趋势和 Flaky 热力图 |

### 功能 1.4：Agent 用例精调对话

> 源文件：`core/agent_service.py` → `refine_testcase_with_agent()`
> 前端：`AgentRefineDialog.vue`

#### 现状

- 单用例维度的 AI 对话精调，打开时自动让 Agent 分析并提出改进建议
- Agent 返回 `suggestions` 标签列表，用户可点击标签快速反馈
- `action=update` 时 Agent 直接修改用例字段（name, steps, expected_result 等）并 bump version
- **XSS 风险**：`v-html="renderMarkdown(msg.updatedTestcase.markdown_content)"` 使用 `marked.parse()` 输出未经消毒
- **对话无上限**：`conversation_history` 持续增长无截断
- **Markdown 渲染不一致**：Agent 返回的 `markdown_content` 和前端展示的 `marked` 输出样式不统一

#### 升级方案

1. **XSS 防护**：安装 `dompurify`，所有 `v-html` 渲染前执行 `DOMPurify.sanitize()`
2. **对话历史窗口**：超过 20 轮时，将早期对话压缩为摘要前缀
3. **流式响应**：Agent 回复改用 streaming，前端逐字显示回复文字，提升交互体感
4. **Markdown 预览增强**：统一使用 `markdown-body` 样式类，添加代码高亮（highlight.js）

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 安全性 | XSS 风险 | DOMPurify 消毒，安全 |
| 对话体感 | 漫长等待后一次性展示 | 逐字流式输出 |
| Token 成本 | 无控制，线性增长 | 窗口截断 + 摘要压缩 |
| 渲染质量 | 纯文本，无代码高亮 | 完整 Markdown 渲染 + 代码高亮 |

### 功能 1.5：代码仓库分析

> 源文件：`core/repo_analyzer.py`
> 前端：`CodeAnalysisPanel.vue`

#### 现状

- 支持 CLI（Claude Code CLI）和 SDK 两种分析引擎
- SDK 模式手动搜索路由关键词 → 收集代码片段 → 调用 Claude API
- CLI 模式直接让 Claude CLI 在仓库目录下分析
- 分析结果按功能点分组，包含页面路由（含元素选择器）和 API 端点（含参数和响应字段）
- **心跳卡住检测**：前端已实现 30s 无心跳显示"分析中断"，但无自动恢复
- **分析结果不可编辑**：AI 分析出的功能模块/元素/参数用户无法手动修正
- **无增量分析**：代码变更后需完全重新分析

#### 升级方案

1. **分析结果可编辑**：允许用户修改 AI 识别出的元素选择器、API 参数、功能分组，修正 AI 的误识别
2. **增量分析**：记录上次分析的 commit hash，仅对变更文件重新分析，合并结果
3. **分析质量评估**：对每个发现的页面/API 标注置信度（基于元素数量、代码匹配度），低置信度的标记需人工确认

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 结果准确性 | AI 说了算，错误需接受 | 用户可修正误识别 |
| 分析效率 | 每次全量分析 | 增量分析，变更部分秒级完成 |
| 可信度 | 无评估 | 置信度标注，低置信度需确认 |

### 功能 1.6：Agent 自动执行测试

> 源文件：`core/agent_service.py` → `AgentRunner`
> 前端：`ExecutionObserver.vue`（观察面板）

#### 现状

- `AgentRunner` 实现 Claude tool_use 多轮循环：发送消息 → Claude 返回 tool_use → 执行工具 → 回传结果 → 继续
- 每个工具执行后自动截图并持久化到 `media/{execution_id}/step_{n}.png`
- 通过 WebSocket 实时推送步骤事件（step_start, step_complete, execution_end）
- **无取消机制**：Agent 开始后无法中断，只能等服务端重启
- **无整体超时**：`max_turns` 限制轮次，但单轮可耗时极长，整体无时长上限
- **全量快照 Token 开销大**：每次浏览器工具执行后都返回完整 DOM 快照，复杂页面 2000+ token/次
- **Claude API 无重试**：网络超时、429 限速直接崩溃，浏览器资源和进度全丢

#### 升级方案

1. **取消执行**：增加 `cancel()` 方法和 `POST /api/executions/{id}/cancel/` 端点，每轮循环检查 cancel 事件；前端观察页增加"取消执行"按钮
2. **整体超时保护**：默认 10 分钟总超时，超时后返回 `status: 'timeout'`
3. **精简快照策略**：工具执行后默认返回精简快照（仅交互元素，< 500 token），仅当 Claude 主动调用 `browser_snapshot` 时返回完整快照
4. **API 重试**：对 Claude API 调用增加指数退避重试（3 次），超时和 500 错误自动重试

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 可控性 | 启动后不可控 | 随时可取消，有总超时保护 |
| Token 消耗 | 20 轮约 40000+ token（仅快照） | 降低 60%+（精简快照策略） |
| 稳定性 | 网络抖动直接崩溃 | 自动重试，用户无感知 |

## 二、UI / 交互优化（12 个功能点）

### 功能 2.1：仪表盘（Dashboard）

> 源文件：`frontend/src/views/Dashboard.vue`

#### 现状

- 4 个统计卡片：项目数、用例数、执行次数、AI 对话数
- 快捷操作按钮：新建项目、执行方案、查看执行记录、脚本管理
- 左右两栏：最近项目列表 + 最近执行列表（各 5 条，分页）
- **无通过率统计**：统计卡片缺少"通过率"这个核心 QA 指标
- **无趋势图**：没有时间维度的执行趋势，无法看到测试健康度变化
- **最近执行无跳转**：执行列表中点击记录无操作入口

#### 升级方案

1. **增加通过率卡片**：第 5 个卡片显示最近 7 天通过率（passed / total），使用环形进度图
2. **执行趋势迷你图**：在"最近执行"卡片上方添加 7 天执行次数 + 通过率折线图（使用轻量图表库如 `vue-sparklines` 或纯 CSS 实现）
3. **执行记录可点击**：点击执行记录直接跳转到观察页面或详情弹窗

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 信息密度 | 4 个基础数字 | 5 个指标含通过率 + 趋势图 |
| 可操作性 | 最近执行不可点击 | 点击直接跳转观察/详情 |
| QA 感知 | 无健康度概念 | 通过率趋势一目了然 |

### 功能 2.4：项目详情页 — 代码分析向导

> 源文件：`frontend/src/views/ProjectDetail.vue` → 代码分析 tab + `CodeAnalysisPanel.vue`

#### 现状

- 5 步向导：拉取仓库 → 代码分析 → 选择目标 → 生成用例 → 确认保存
- 分析结果按功能点折叠展示，支持展开查看页面元素/API 参数
- 支持全选/分组选择目标
- **分析中无阶段提示**：只显示"正在分析仓库代码"，无具体阶段（搜索路由中/分析代码中/生成结果中）
- **分析失败恢复差**：失败后只能重新分析，不能在失败基础上调整
- **选择目标无搜索**：分析出 50+ 个目标时难以找到特定的页面/API

#### 升级方案

1. **分析阶段提示**：后端通过 WebSocket 推送分析阶段（`searching_routes` / `reading_code` / `generating_results`），前端显示对应的阶段文案和进度
2. **目标搜索**：在选择目标步骤添加搜索框，按名称/路径过滤
3. **失败部分恢复**：分析失败时，如果已有部分结果（`discovered_items` 非空），提示用户"是否保留已有结果并重新分析剩余部分"

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 等待体验 | 黑盒等待 | 阶段性进度提示 |
| 目标选择 | 手动翻找 | 搜索即时过滤 |
| 失败恢复 | 全部重来 | 部分保留 + 重试 |

### 功能 2.5：执行观察面板

> 源文件：`frontend/src/views/ExecutionObserver.vue` + `BrowserView.vue` + `ExecutionTimeline.vue` + `ToolDetailPanel.vue`

#### 现状

- 左侧浏览器截图画面（`BrowserView`），右侧步骤时间线（`ExecutionTimeline`）
- 支持画中画模式、全屏模式
- WebSocket 实时接收步骤事件，REST 回填断线期间丢失的步骤
- 截图切换使用 `<transition name="fade" mode="out-in">`
- **截图切换闪烁**：out-in 模式导致旧图淡出后新图才淡入，中间短暂黑闪
- **无自动滚动**：新步骤到来时时间线不自动滚到底部
- **无整体进度条**：不知道总共执行多少步、当前在第几步
- **WebSocket 断线无 UI 提示**：重连失败后用户看不到任何提示
- **步骤详情面板信息不足**：`ToolDetailPanel` 只显示 target、result、screenshot，不显示工具输入参数

#### 升级方案

1. **交叉淡入淡出**：将截图切换改为 crossfade 效果（新图叠在旧图上方淡入），消除黑闪
2. **自动滚动 + 进度条**：新步骤到来时自动滚到底部（smooth scroll）；顶部增加 `el-progress` 进度条（基于当前步数/总步数估算）
3. **WebSocket 断线提示**：断线重连失败时显示 `el-alert` 错误提示 + "重试"按钮
4. **步骤详情增强**：`ToolDetailPanel` 增加工具输入参数展示（如填写的值、点击的选择器），结果文本增加展开/收起和一键复制

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 视觉体验 | 截图切换黑闪 | 丝滑交叉淡入淡出 |
| 进度感知 | 不知道在哪一步 | 进度条 + 自动滚动 + 步骤计数 |
| 断线感知 | 无提示 | 醒目的断线提示 + 重试 |
| 调试能力 | 只看到 target 和 result | 完整的输入参数和输出结果 |

### 功能 2.6：执行回放

#### 现状

| 维度 | 升级前 | 升级后 |

## 三、核心功能优化（5 个功能点）

### 功能 3.1：脚本转换（执行记录 → 可回放脚本）

> 源文件：`core/script_converter.py`

#### 现状

- 从 `ExecutionRecord` 的 `step_logs` / `tool_calls_log` 中提取浏览器操作步骤
- 自动将用户输入值提取为动态参数（`{{param_name}}` 模板语法）
- 支持参数去重、断言选择器关联
- **参数去重逻辑有 bug**：`_extract_param_value_from_step` 匹配到模板字符串时返回空字符串，与"未找到"返回值相同，导致 `_deduplicate_param_names` 始终认为值相同，整个去重逻辑失效
- **日志 JSON 解析脆弱**：通过 `=== TOOL_CALLS_JSON ===` 标记提取 JSON，日志后可能附加额外文本导致解析失败

#### 升级方案

1. **修复参数去重**：使用唯一哨兵值区分"找到模板"和"未找到"
2. **增强 JSON 解析**：使用 `json.JSONDecoder().raw_decode()` 只解析有效 JSON 部分，忽略尾部附加文本
3. **转换结果校验**：转换完成后自动校验脚本结构完整性（步骤数 > 0、参数引用完整），失败时给出具体错误提示

#### 升级后效果

| 维度 | 升级前 | 升级后 |
|------|--------|--------|
| 转换质量 | 参数去重失效，同名不同值的参数被合并 | 正确区分同名不同值参数 |
| 解析稳定性 | 日志有附加文本就解析失败 | 容错解析，忽略尾部噪声 |
| 用户反馈 | 转换失败只报"转换失败" | 具体错误原因提示 |

### 功能 3.2：脚本回放执行

> 源文件：`core/script_executor.py`

#### 现状

| 灵活性 | 断言失败即停 | 可选继续执行 |
