## 二、Agent 系统深度分析

> 对 `agent_service.py`、`agent_tools.py`、`execution_engine.py`、`script_converter.py`、`script_executor.py`、`consumers.py`、`event_emitter.py`、`screenshot_stream.py` 逐文件分析。

### 2.1 AgentRunner 核心循环 (`agent_service.py`)

#### 2.1.1 步骤持久化竞态条件 (Critical)

**现状**：`_persist_step` 先在主线程读取 `record.step_logs` 并追加，再在新线程中保存。`t.join(timeout=5)` 超时后主线程继续读旧数据，而旧线程仍在写入——步骤可能丢失。

**升级方案**：
```python
_step_lock = threading.Lock()

def _persist_step(self, step):
    with self._step_lock:
        record = ExecutionRecord.objects.get(pk=self._execution_id)
        current_logs = list(record.step_logs or [])
        current_logs.append(step)
        record.step_logs = current_logs
        record.save(update_fields=['step_logs', 'tool_calls_count'])
```

#### 2.1.2 Claude API 调用无重试

**现状**：`run()` 中 `client.messages.create()` 无 try/except。网络超时、429 限速、500 错误会直接崩溃，浏览器资源和进度全丢。

**升级方案**：
```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(min=2, max=30))
def _call_claude(client, **kwargs):
    return client.messages.create(**kwargs)
```

#### 2.1.3 会话历史无限增长

**现状**：`refine_testcase_with_agent` 中 `conversation_history` 无上限，多轮精炼后超 JSONField 限制且浪费 token。

**升级方案**：保留最近 10 轮对话，超出截断。

#### 2.1.4 工具调用日志截断

**现状**：`result_text[:5000]` 静默截断。复杂页面快照截断后数据无用。

**升级方案**：截断时附加标记 `result_text[:5000] + '\n...[已截断, 完整长度: %d]' % len(result_text)`。

### 2.2 浏览器工具 (`agent_tools.py`)

#### 2.2.1 截图路径遍历风险

**现状**：`_execute_browser_screenshot` 中 `save_path` 参数由 Claude 生成，无路径校验。

**升级方案**：
```python
def _validate_screenshot_path(save_path):
    media_root = os.path.abspath(settings.MEDIA_ROOT)
    abs_path = os.path.abspath(save_path)
    if not abs_path.startswith(media_root):
        raise ValueError("截图路径必须在 MEDIA_ROOT 下")
```

#### 2.2.2 全量快照 Token 开销过高

**现状**：每个浏览器工具执行后都调用 `_snapshot_page(page)` 返回完整 DOM 树。复杂页面 2000+ token/次，20 轮执行仅快照就消耗 40000+ token。

**升级方案**：
- 默认返回精简快照（仅交互元素，最大 500 token）
- 仅当 Claude 主动调用 `browser_snapshot` 时返回完整快照

#### 2.2.3 URL 变化检测使用固定 sleep

**现状**：`browser_click` 中 `time.sleep(0.3)` 检测 URL 变化。SPA 可能 50ms 就完成跳转，慢页面可能需要 2s。

**升级方案**：使用 Playwright 原生 `page.wait_for_url(timeout=500)` 替代固定 sleep。

#### 2.2.4 `get_tools_for_context` 忽略 `has_context` 参数

**现状**：无论是否有预提取的代码上下文，Agent 都只提供浏览器工具。缺少上下文时 Agent 无法主动查阅源码。

**升级方案**：无上下文时额外提供 repo 探索工具（`list_files`、`read_file`）。

#### 2.2.5 `report_result` 缺少状态值校验

**现状**：不校验 `status` 字段，Claude 返回 `'success'` 会被原样存储。

**升级方案**：限制 `status` 只接受 `passed/failed/error`，其余默认为 `passed`。

#### 2.2.6 JS 代码管理问题

**现状**：7 个大型 JS 模板字符串内嵌在 1300 行 Python 文件中，难以维护和调试。

**升级方案**：提取为独立 `.js` 文件到 `core/agent/js/` 目录，运行时通过 `pathlib` 读取。

### 2.3 脚本转换器 (`script_converter.py`)

#### 2.3.1 参数去重系统失效

**现状**：`_extract_param_value_from_step` 匹配到模板字符串时返回 `''`，与"未找到"返回值相同。导致 `_deduplicate_param_names` 始终看到 `values = ['', '']`，整个参数去重逻辑无效。

**升级方案**：使用唯一标记区分"找到模板"和"未找到"：
```python
class _Found:
    def __bool__(self): return True
SENTINEL = _Found()
```

#### 2.3.2 日志 JSON 解析脆弱

**现状**：通过 `=== TOOL_CALLS_JSON ===` 标记从日志提取 JSON，日志后可能附加额外文本导致解析失败。

**升级方案**：使用 `json.JSONDecoder().raw_decode()` 只解析有效 JSON 部分。

### 2.4 脚本执行器 (`script_executor.py`)

#### 2.4.1 浏览器管理代码重复

**现状**：`_ensure_browser` 和 `_cleanup_browser` 从 `agent_service.py` 复制粘贴，约 80 行重复代码。

**升级方案**：提取共享的 `BrowserManager` 类到 `core/browser_manager.py`。

#### 2.4.2 单步执行无超时保护

**现状**：`browser_batch_action` 可含多个子操作且无整体超时限制。

**升级方案**：在 `run()` 方法中为每个步骤添加 120s 整体超时。

### 2.5 WebSocket 消费者 (`consumers.py`)

#### 2.5.1 心跳递归创建线程

**现状**：每次心跳创建新 `threading.Timer`（=新线程），长连接下累积大量短命线程。

**升级方案**：改用单线程 + `threading.Event.wait(timeout=INTERVAL)` 模式。

#### 2.5.2 `send()` 无异常保护

**现状**：`step_event` 和 `frame_event` 中直接 `self.send()` 无 try/except，客户端断开后抛异常。

**升级方案**：包裹 try/except，记录 debug 日志。

#### 2.5.3 两个消费者代码重复

**现状**：`ExecutionConsumer` 和 `FrameConsumer` 的心跳、连接、断连逻辑几乎相同，约 100 行重复。

**升级方案**：创建 `BaseConsumer` 基类。

### 2.6 截图流 (`screenshot_stream.py`)

#### 2.6.1 内存泄漏风险

**现状**：`_latest_frames` 字典中的条目在 Agent 崩溃时不会被清理。

**升级方案**：添加 TTL 清理机制，超过 1 小时的帧自动清除。

#### 2.6.2 Watchdog 无客户端时仍发心跳

**现状**：`_FrameWatchdog` 即使没有客户端连接也持续发送心跳事件。

**升级方案**：增加连接计数器，无客户端时暂停发送。

### 2.7 Agent 横向功能增强

#### 2.7.1 Agent 执行取消机制

**现状**：Agent 开始执行后无法从外部取消，用户只能等待完成或重启服务。

**升级方案**：
```python
class AgentRunner:
    def __init__(self, ...):
        self._cancel_event = threading.Event()

    def cancel(self):
        self._cancel_event.set()

    def run(self, ...):
        for turn in range(max_turns):
            if self._cancel_event.is_set():
                return {'status': 'cancelled', ...}
```
前端增加"取消执行"按钮，调用 `POST /api/executions/{id}/cancel/`。

#### 2.7.2 Agent 执行超时保护

**现状**：`max_turns` 限制轮次，但单轮可能耗时极长。无整体执行时长限制。

**升级方案**：默认 10 分钟总超时，超时后返回 `status: 'timeout'`。

#### 2.7.3 Agent 中间件/钩子机制

**现状**：工具执行前后无统一拦截点，日志、监控逻辑散落各处。

**升级方案**：
```python
class AgentMiddleware:
    def before_tool(self, tool_name, tool_input, context): ...
    def after_tool(self, tool_name, tool_input, result, context): ...

class LoggingMiddleware(AgentMiddleware): ...
class MetricsMiddleware(AgentMiddleware): ...
```

## 三、前端 UI 交互深度分析

### 3.1 截图流改进 — 增加操作前截图

**现状**：`agent_service.py` 第 555 行注释明确"浏览器工具执行后自动截图"。`_auto_screenshot` 仅在工具执行完毕后调用，保存为 `step_{n}.png`。用户在回放/观察时只能看到操作后的页面状态，无法直观对比操作前后变化。

截图已持久化到 `media/{execution_id}/` 目录，不存在丢失问题。但仅有操作后截图，缺少操作前截图。

**升级方案**：在每个浏览器工具执行前增加一次截图，保存为 `step_{n}_before.png`：

```python
# agent_service.py — 工具执行前
before_screenshot_path = ''
if tool_name in browser_tool_names and context.get('page') and not context['page'].is_closed():
    before_screenshot_path = self._auto_screenshot(context, step_num, suffix='_before')

# 执行工具...
result_text = _execute_tool(...)

# 执行后截图（现有逻辑不变）
auto_screenshot_path = self._auto_screenshot(context, step_num)
```

`_auto_screenshot` 方法增加 `suffix` 参数：
```python
save_path = os.path.join(screenshot_dir, f'step_{step_num}{suffix}.png')
```

前端 `step_logs` 增加 `screenshot_before_path` 字段，观察页支持前后对比查看。

**存储成本**：每步增加一张图约 100-300KB，20 步执行增加 2-6MB，完全可接受。

### 3.2 执行观察页面 (`ExecutionObserver.vue`)

#### 3.2.1 截图切换闪烁

**现状**：`<transition name="fade" mode="out-in">` 导致截图切换时旧图淡出后新图才淡入，中间短暂黑屏。

**升级方案**：改为交叉淡入淡出，消除黑闪：
```vue
<transition name="crossfade">
  <img :key="frameKey" :src="frameSrc" />
</transition>
```

#### 3.2.2 步骤时间线交互缺失

**现状**：
- 无自动滚动到最新步骤
- 无步骤搜索/筛选
- 执行中 vs 卡住无法区分（都是蓝色边框，无动画）
- 无步骤耗时显示
- 无整体执行进度条

**升级方案**：
```vue
<!-- 自动滚动 + 进度条 -->
<el-progress :percentage="progressPercent" :status="progressStatus" />

<script setup>
watch(steps, (newSteps) => {
  if (newSteps.length > 0) {
    nextTick(() => {
      const container = timelineRef.value?.$el
      container?.scrollTo({ top: container.scrollHeight, behavior: 'smooth' })
    })
  }
})
</script>
```

#### 3.2.3 WebSocket 重连失败无 UI 提示

**现状**：`useExecutionObserver` 达到最大重连次数后设置 `error` ref，但 UI 不显示。

**升级方案**：
```vue
<el-alert v-if="wsError" type="error" :closable="false" show-icon>
  {{ wsError }}
  <el-button @click="manualReconnect" size="small">重试</el-button>
</el-alert>
```

### 3.3 项目详情页 (`ProjectDetail.vue`)

#### 3.3.1 巨型组件问题

**现状**：~690 行单文件包含概览、测试用例树、代码分析向导（5步）、AI 生成、Agent 执行、抽屉和对话框。

**升级方案**：按功能域拆分：
```
views/ProjectDetail/
├── index.vue                   # 主布局 + tab 切换
├── OverviewTab.vue             # 项目概览
├── TestCaseTreeTab.vue         # 测试用例树

    └── AnalysisWizard.vue      # 分析向导步骤
```

#### 3.3.2 功能树无虚拟滚动

**现状**：500+ 测试用例时 DOM 节点过多，渲染卡顿。

**升级方案**：使用 `el-tree-v2`（Element Plus 虚拟化树）。

### 3.4 执行记录页 (`Executions.vue`)

#### 3.4.1 运行中执行无自动刷新

**现状**：执行中的记录不会自动更新状态，需手动刷新页面。

**升级方案**：对运行中的执行进行 5 秒轮询，状态变化后停止轮询。

#### 3.4.2 操作按钮过多，移动端拥挤

**升级方案**：使用 `el-dropdown` 折叠次要操作：
```vue
<el-dropdown trigger="click">
  <el-button size="small">更多 <el-icon><ArrowDown /></el-icon></el-button>
  <template #dropdown>
    <el-dropdown-menu>
      <el-dropdown-item @click="handleConvert(row)">转为脚本</el-dropdown-item>
      <el-dropdown-item @click="handleReplay(row)">重放</el-dropdown-item>
      <el-dropdown-item @click="handleDelete(row)" divided>删除</el-dropdown-item>
    </el-dropdown-menu>
  </template>
</el-dropdown>
```

### 3.5 脚本编辑器 (`ScriptEditorView.vue`)

#### 3.5.1 参数删除不可逆

**现状**：`deleteParam` 会将所有步骤中的 `{{key}}` 替换为默认值并永久删除参数，无确认对话框。

**升级方案**：增加 `ElMessageBox.confirm` 确认提示。

### 3.6 工具详情面板 (`ToolDetailPanel.vue`)

**现状**：仅显示 target、result、screenshot，不显示工具输入参数。

**升级方案**：增加输入参数展示 + 结果展开/复制功能。

### 3.7 执行回放 (`ExecutionReplay.vue`)

**现状**：`_rebuildToStep` 在每次滑块值变化时重建所有可见步骤，O(n) 操作导致拖动卡顿。

**升级方案**：使用 `computed` 切片替代全量重建。增加键盘控制（空格播放/暂停，方向键步进）。

### 3.8 跨组件性能问题

| 问题 | 影响 | 升级方案 |
|------|------|---------|
| 所有列表无虚拟滚动 | 大量数据时性能下降 | 使用 `el-tree-v2`、`el-table-v2`、`vue-virtual-scroller` |
| Deep Watch on steps 数组 | 每步属性变化都触发重计算 | 改为仅监听 `steps.value.length` |
| 过滤器/搜索框无防抖 | 每次输入直接触发 API 请求 | 使用 `useDebounceFn` 防抖 |
| 前端 Bundle 全量引入 Element Plus | 包体积大 | 使用 `unplugin-vue-components` 按需引入 |

### 3.9 XSS 防护

**现状**：前端使用 `marked` 库渲染 AI 返回的 Markdown，未配合 DOMPurify 做消毒。`AgentRefineDialog` 中 `v-html` 直接渲染 `marked.parse()` 的输出。

**升级方案**：
```bash
npm install dompurify
```
```javascript
import DOMPurify from 'dompurify'
const safeHtml = DOMPurify.sanitize(marked(rawMarkdown))
```

## 四、核心数据流与关系链分析

### 4.1 核心数据流（6 条链路）

#### 链路 1：代码分析 / 手动创建 → 测试用例

```
代码分析:  RepoAnalysis.discovered_items → 用户勾选 → POST /batch-generate/
手动创建:  ProjectDetail → "新建用例" → POST /testcases/
```

**交互问题**：
- 代码分析到测试用例是异步批量生成，无进度条
- 手动创建和 AI 生成的用例混在一起，`created_by` 字段未在 UI 中显示

#### 链路 2：测试用例 → Agent 执行

```
单条: TestCase → POST /agent-execute/ → ExecutionRecord(testcase=tc)
批量: Project  → POST /execute-agent/ → 多条 ExecutionRecord(testcase=null)
分组: group    → POST /execute-group/ → 多条 ExecutionRecord
```

**交互问题**：
- **批量执行时 `testcase=null`**：无法关联回具体用例，用户看到"批量执行"但不知道包含哪些
- **单条执行后无跳转**：ProjectDetail 点击"执行"后仅 toast 提示，不跳转到观察页
- **执行结果不回写用例状态**：Agent 完成后 `TestCase.status` 不自动更新

#### 链路 3：Agent 执行 → 操作记录

```
AgentRunner.run() → ExecutionRecord(step_logs=[], screenshots=[], replay_script={})
```

**交互问题**：
- 执行完成后无"下一步"引导（如"转为脚本"、"重新执行"）
- 操作记录列表不显示关联的测试用例名称

#### 链路 4：操作记录 → 脚本（双脚本系统）

| 存储 | 位置 | 用途 |
|------|------|------|
| `ExecutionRecord.replay_script` | JSON 字段 | 嵌入执行记录内的草稿脚本 |
| `Script` 模型 | 独立数据库表 | 可管理、可搜索的正式脚本 |

**核心问题**：`replay_script` 始终存在（Agent 执行完自动生成），用户可能困惑"为什么执行记录里已经有脚本了还要转换"。

**建议**：统一为单一流程：
1. Agent 执行完成 → 自动生成草稿脚本（`replay_script`）
2. 观察页 → "查看脚本" → 打开草稿编辑器
3. 草稿编辑满意后 → "保存为正式脚本" → 创建 `Script` 记录
4. 脚本列表页仅显示正式脚本

在 UI 上通过标签区分：
```vue
<el-tag v-if="isDraft" type="warning">草稿脚本</el-tag>
<el-tag v-else type="success">正式脚本</el-tag>
```

#### 链路 5：脚本 → 回放

```
Script → POST /scripts/{id}/execute/ → ExecutionRecord(source_execution=原记录, mode='replay')
```

**交互问题**：
- 回放产生的 `ExecutionRecord` 通过 `source_execution` 指向原始记录，但 UI 上**不可从原始记录跳转到回放记录**
- 脚本编辑器（`/executions/{id}/script`）和脚本详情页（`/scripts/{id}`）是两个页面编辑同一份数据，可能产生混淆

#### 链路 6：测试方案 → 执行

```
TestPlan → POST /execute/ → PlanExecution → 多条 ExecutionRecord(plan_execution=pe)
```

**交互问题**：
- 方案执行结果页显示汇总，但**不能从单条执行记录跳转到所属方案**

### 4.2 导航缺口（10 条缺失的关联跳转）

| # | 起点 | 终点 | 建议 |
|---|------|------|------|
| 1 | 执行记录 → 测试用例 | 项目详情定位用例 | 点击用例名跳转 |
| 2 | 执行记录 → 项目 | 项目详情 | 添加项目名列 |
| 3 | 执行记录 → 脚本 | 正式脚本详情 | 区分草稿/正式 |
| 4 | 执行记录 → 回放记录 | 回放历史列表 | 原始记录下显示标签 |
| 5 | 执行记录 → 所属方案执行 | 方案执行详情 | 添加链接 |
| 6 | 测试用例 → 最近执行 | 执行历史 | 添加状态指示 |
| 7 | 测试用例 → 关联脚本 | 脚本详情 | 转换后自动关联 |
| 8 | 脚本 → 原始执行记录 | 执行详情 | 添加来源链接 |
| 9 | 脚本 → 回放历史 | 执行历史 | 添加标签页 |
| 10 | 项目 → 最近执行 | Dashboard 已有 | 已实现 ✓ |

### 4.3 关键交互改进

#### 4.3.1 执行完成后"下一步"引导

**现状**：Agent 执行完成后，观察页仅显示结果状态。

**升级方案**：
```vue
<div v-if="isTerminalStatus" class="next-steps-panel">
  <el-result :icon="resultIcon" :title="resultTitle">
    <template #extra>
      <el-button type="primary" @click="convertToScript">转为可编辑脚本</el-button>
      <el-button @click="rerunExecution">重新执行</el-button>
      <el-button @click="goToTestcase">返回测试用例</el-button>
    </template>
  </el-result>
</div>
```

#### 4.3.2 执行结果回写用例状态

**现状**：Agent 执行完成后，`TestCase.status` 不自动更新。

**升级方案**：在 `AgentRunner.run()` 末尾，根据执行结果更新关联用例状态（`passed`/`failed`）。

#### 4.3.3 批量执行关联用例

**现状**：批量执行时 `ExecutionRecord.testcase = null`，无法追溯。

**升级方案**：逐条创建 `ExecutionRecord` 并关联对应 `TestCase`。

#### 4.3.4 用例卡片增强

**现状**：用例卡片仅显示名称、优先级、步骤。

**升级方案**：添加最近执行状态指示 + 快速操作（执行/历史/脚本）。

#### 4.3.5 面包屑导航增强

**现状**：仅显示一级父路由，不显示关联的项目/用例信息。

**升级方案**：路由 meta 中携带上下文（项目名 → 用例名 → 执行记录）。
